# 0.4 Parallelization Guide — split the bottlenecks before the fan-out

**Status:** planning doc (read-only analysis, no source changed). Produced on `feature/0.4-parallel-guide`.
**Scope:** 0.4 = "the living creatures + their entire life cycle." Nearly every 0.4 workstream converges on
the same handful of files (`Creature.gd`, `EcologyService.gd`, the cognition pair, the reaction registry).
That convergence is the thing to design out **before** launching agents — otherwise a wide fan-out collapses
into a queue on those files.

---

## How to use this guide (fresh-session quickstart)

1. **Read the parallelizability-first rule** (CLAUDE.md → "PARALLELIZABILITY is a first-class refactor
   driver" + the `parallelizability-first-refactor` memory): before fanning out, look at which files each
   unit touches; if several route through ONE file, split that file FIRST into single-owner modules.
2. **Do Phase 0 first, serialized, one owner at a time** (§4). Phase 0 is *seam-directed* — extract only the
   seams the 0.4 fan-out needs, not a speculative teardown of every hub. Each Phase-0 split is a small,
   mechanical, behavior-preserving extraction verified with a short `SIM_REPORT` run (`--run-frames=120`).
3. **Then fan out** using the disjoint-ownership groups in §5. Give each agent one module to own plus the
   shared-interface contract it must honor. Freeze the two shared registries (cognition-core interface,
   reaction channel enum) before their dependent agents launch.
4. **Honor the extract-only hubs:** `VoxelWorld.gd` and `MaterialField3D.gd` get *wiring only* (one-line
   `add_child` / delegate), never new behavior. Same discipline now applies to `Creature.gd` and
   `EcologyService.gd` once split — they become thin cores that delegate to the new modules.

### Caveats / uncertainty (be honest about these)
- **The `## 0.4 — THE LIVING CREATURES` section is *not* in this branch's `TODO.md`.** The workstream list
  below is taken from the 0.4 task brief. Reconcile against `TODO.md` once the 0.4 section lands; the file
  mapping here is derived from the actual current code, so the split plan holds regardless.
- **`EcologySpawner.gd` does not exist yet** — only `ecology/EcologyService.gd` (730 lines) does. The split
  in §3 *creates* the spawner.
- **`MaterialFungus3D.gd` / `MaterialScent3D.gd` are gone** (only `.uid` stubs remain) — scent/fungus are now
  GPU-only channels in the cubed-sphere reaction kernel, not CPU modules. Soil/nitrogen work lands as
  *reaction records + a channel slot*, not a new CPU channel module.
- **Fish have no cognition today** (`Fish.gd` `_physics_process` is purely reactive) and **disease has no
  code at all** — those two are greenfield, not refactors.

## 3. Seam-directed split plan (per bottleneck)

Extract the concern → a named owner-module, so each 0.4 workstream edits *its own* module, not the hub.
Hubs stay thin (delegate/forward only).

### 3.1 `Creature.gd` → concern modules (pattern already established in `actors/creature/`)
The dir already holds `CreatureLeadership/Metabolism/Flocking/Think/Senses/Nesting/Ragdoll/Thirst/…`. Add:

| New module | Extract from `Creature.gd` | Owner |
|---|---|---|
| `CreatureDigestion.gd` | `energy`/`max_energy`/`metabolism` bookkeeping, `feed()`/`nutrition()`/`food_profile()` energy-side, + fold in `CreatureMetabolism.gd`; add gut/microbiome state | **W1** |
| `CreatureExcretion.gd` | `_deposit_waste()`, `_poop_cd`/`_urine_cd`, the waste-emission portion of `_physics_process` | **W1** |
| `CreatureLifeStage.gd` | `age`/`max_age`/`maturity_age`, `is_mature()`, growth-scale-with-age, aging→`die("old_age")` routing | **W6** |
| `CreatureReproduction.gd` | a thin `mating_ready()` / `on_bred()` hook that `EcologyBreeding` calls (mating logic lives ecology-side; the per-creature readiness/cooldown lives here) | **W6** |
| `CreatureCompanion.gd` | bond/training/recall state; interplay with `hold_begin/end`/`throw` (the player's "hand") and cognition rewards | **W4** |

After extraction `Creature.gd` keeps: node lifecycle (`_process`/`_physics_process` orchestration), movement
integration, the `setup()` wiring that instantiates + injects these modules. Treat it as **extract-only** from
here — new life-cycle behavior goes in the modules.

### 3.2 `EcologyService.gd` → split the god-object
| New module | Extract | Owner |
|---|---|---|
| `EcologySpawner.gd` | `spawn`/`_instance_actor`/`_place_on_surface`/`_random_spawn_point`/`spawn_initial`/`populate_environment`/`_orient_to_surface` | infra (shared, low churn) |
| `EcologyBreeding.gd` | `_tick_breeding`/`_breed_genome`/`_inherit_nest`/`spawn_nest`/`broadcast_call` | **W6** |
| `EcologyPlants.gd` | `_tick_plant_seeding`/`seed_plant_at`/`_can_grow_here`/`_plant_config` | **W2** |
| `EcologyAquatic.gd` | `stock_initial_aquatic`/`_tick_aquatic`/`_random_aquatic_point*`/`_aquatic_depth` | **W5/W6** (fish) |
| `EcologyPollination.gd` (new, later) | flower→bee pollination tick | **W3** |

`EcologyService.gd` keeps: setup/wiring, the `_physics_process` tick dispatcher that calls each sub-ticker,
seismic/fire broadcasts. Thin facade after the split.

### 3.3 Cognition → generalize off `LACreature` (coupling seam, not a line split)
- Define a minimal **cognizer interface** (duck-typed) that `Cognition.decide(c,…)` and
  `CognitionScheduler.request(...)` require: `situation_signature()`, `innate_action()`, `apply_action(a)`,
  `reinforce_cue()/cue_value()` (already on `Creature`). Document it once; freeze it.
- Extract the creature-specific perception scans in `CognitionScheduler` (`_gather_context`,
  `_scan_predators/_scan_prey/_scan_group_visible`) behind that interface so a Fish/Bee can supply its own.
- New per-species adapters live under `cognition/adapters/`: `FishCognition.gd` (**W5**),
  `BeeCognition.gd` (**W3**). Companion training (**W4**) reuses the existing `reinforce_cue` reward path.

### 3.4 Reaction registry → grouped record providers (for W1 + W8)
- `MaterialReactions3D.records()` is one function returning `_rec(...)` records. Split authoring into grouped
  providers (e.g. `material/reactions/{WaterPhaseReactions,CombustionReactions,BiologyReactions,
  SoilNitrogenReactions}.gd`), each returning its slice; `records()` concatenates them. W1 then owns
  `SoilNitrogenReactions.gd` and W8 owns a composition provider without editing the same function.
- **Unavoidable shared edit:** the channel enum (`TEMP..WINDSPEED`) + the GPU `read_ch/add_ch` switch-ladder in
  `reactions_sphere3d.glsl`. Adding `NITROGEN` (W1) and any composition channel (W8) both touch these two
  spots. **Batch all 0.4 channel additions into one serialized pass** (single owner) before the reaction
  agents fan out — see Phase 0.4.

### 3.5 `Plant.gd` → `PlantLifeCycle.gd` (bloom/seed), `Fish.gd` → `FishBody.gd` (mesh builders)
- Extract `_apply_growth`/`_grown_fraction`/`has_seed`/`consume`/`is_mature` → `plant/PlantLifeCycle.gd`
  (**W2** owns; **W3** adds bloom→flower variants via config).
- Extract Fish's `_build_*_body`/`_build_model` (~200 lines, `_build_fish/crab/turtle/whale/jellyfish_body`) →
  `actors/FishBody.gd` so **W5** edits Fish cognition/core while the mesh code stays put.

## 4. Phase 0 — serialized split order (do these FIRST, one owner each)

Ordered by parallelism unlocked. Each step: one agent, mechanical behavior-preserving extraction, verify with
`LA_NO_STREAMER=1 … VoxelWorld.tscn -- --run-frames=120` (SIM_REPORT aggregates unchanged), commit, next.

| Step | Split | Unlocks | Why this order |
|---|---|---|---|
| **P0.1** | `Creature.gd` → Digestion / Excretion / LifeStage / Reproduction / Companion modules (§3.1) | **W1, W4, W6** (and clears the field for W7) | #1 bottleneck; 4 workstreams blocked on one file. Highest single-step leverage. |
| **P0.2** | `EcologyService.gd` → Spawner / Breeding / Plants / Aquatic (§3.2) | **W2, W6** (breeding), W3 (spawn hook), W5 (aquatic) | #2 bottleneck; unblocks plant + life-arc ecology-side ticks. Independent of P0.1 → *can run concurrently with P0.1* (different file). |
| **P0.3** | Cognition generalized off `LACreature` + `cognition/adapters/` seam (§3.3) | **W3, W4, W5** | Freezes the cognizer interface so bee-learning, fish-cognition, and companion-training fan out in parallel instead of all editing `Cognition.gd`. |
| **P0.4** | Reaction registry → grouped record providers + **one batched channel-enum pass** (§3.4) | **W1, W8** | Turns the record-authoring collision into per-provider ownership; batches the unavoidable enum+kernel edit into a single serialized pass up front. |
| **P0.5** | `Plant.gd` → `PlantLifeCycle.gd`; `Fish.gd` → `FishBody.gd` (§3.5) | **W2, W3, W5** | Small, cheap; lets plant-lifecycle and fish-cognition own their cores cleanly. Can run concurrently with P0.3/P0.4. |

**Concurrency within Phase 0:** P0.1 (Creature) and P0.2 (Ecology) touch different files → launch together.
P0.3, P0.4, P0.5 touch cognition / material / plant+fish respectively → also mutually disjoint, launch
together once P0.1/P0.2 land (P0.2's `EcologyAquatic` and P0.5's `FishBody` both touch Fish-adjacent code but
different files — keep them in the same wave, one owner each). Net: Phase 0 is **two short waves**, not five
serial steps.

**W7 (reusable node) placement — the honest tension.** W7 restructures the very `Creature.gd` core that P0.1
and W1/W4/W6 modules attach to. Two viable orders, pick one explicitly:
- **(a) Frame-first:** fold W7's `CreatureCore` node/sim split into P0.1 as the *frame*, then hang the concern
  modules off it. Cleaner end state, but makes P0.1 bigger/riskier and blocks the whole fan-out on it.
- **(b) Modules-first (recommended):** do P0.1 as mechanical concern extraction (low risk), fan out W1/W4/W6 on
  their modules, and run **W7 last** as a consolidation that re-homes the now-thin core behind a node-agnostic
  interface. Costs one more touch of the modules but keeps the fan-out unblocked and each step small.

## 5. Post-split parallel fan-out

Once Phase 0 lands, group by disjoint file ownership.

### Wave 1 — fully concurrent (disjoint module ownership)
| Group | Workstream | Owns (edits only these) |
|---|---|---|
| A | **W1 nutrient cycle** | `CreatureDigestion.gd`, `CreatureExcretion.gd`, `reactions/SoilNitrogenReactions.gd` |
| B | **W2 plant life cycle** | `plant/PlantLifeCycle.gd`, `ecology/EcologyPlants.gd` |
| C | **W6 life arc** | `CreatureLifeStage.gd`, `CreatureReproduction.gd`, `ecology/EcologyBreeding.gd`, `Genome.gd` |
| D | **W4 companion** | `CreatureCompanion.gd`, `VoxelInteraction.gd` training input, cognition reward hooks |
| E | **W5 fish cognition** | `cognition/adapters/FishCognition.gd`, `Fish.gd` core (post-`FishBody` split) |
| H | **W8 perf/platform** | `MaterialFieldReadback.gd`, `runtime/WebPlatform.gd`, composition channel provider |

These six own non-overlapping modules and can run at once.

### Wave 2 — dependent (launch after their prerequisite lands)
| Group | Workstream | Waits on | Reason |
|---|---|---|---|
| F | **W3 bees + flowers + pollinator selection** | **B** (bloom exists) + **P0.3** (cognition-core) + partly **C** (Genome for pollinator selection) | Bees learn via the cognizer interface; flowers bloom via PlantLifeCycle; pollinator *selection* is heritable-trait pressure on top of W6's reproduction/genome. |
| W7 | **reusable node** | **A/C/D landed** (per §4 order (b)) | Consolidates the now-thin creature core; touches the same modules those groups just wrote. |

### Shared registries to freeze before fan-out (contention that survives the split)
- **Cognizer interface** (P0.3): freeze the method set before D/E/F launch — all three code against it.
- **Reaction channel enum + kernel switch-ladder** (P0.4): W1's `NITROGEN` and W8's composition channels both
  edit the enum + `reactions_sphere3d.glsl`. Add every 0.4 channel in the single P0.4 pass; after that A and H
  only edit their own record-provider files.
- **`data/species/**` JSON**: W3 (bee/flower), W5 (fish), W6 (life-arc traits) all add fields. Different files
  per species → low collision, but agree on the property names (config-over-`if identity`) up front.

## 6. One-paragraph summary for the launcher

Split **`Creature.gd`** and **`EcologyService.gd`** first (two concurrent Phase-0 agents), then generalize the
**cognition pair** and split the **reaction registry** (plus the small Plant/Fish extractions) in a second
Phase-0 wave. That converts the four serialization bottlenecks into single-owner modules so 0.4's eight
workstreams fan out as six concurrent Wave-1 agents (nutrient, plant-cycle, life-arc, companion, fish-cog,
perf/platform) plus two dependent Wave-2 agents (bees/flowers after bloom+cognition-core; reusable-node last).
Freeze the cognizer interface and batch all channel-enum additions before launching, and keep every hub
(`VoxelWorld`, `MaterialField3D`, and now the thinned `Creature`/`EcologyService`) extract-only.
